# 난이도 구조 설명

> 초/중/고 난이도가 코드에서 어떻게 구현돼 있는지, 비전공자도 이해할 수 있게 정리한 문서입니다.

---

## 0. 먼저 짚고 갈 것 — "AI"라는 이름

코드와 UI에서 상대를 "AI"라고 부르지만, **사실 진짜 AI가 아닙니다.**

- 머신러닝 모델 없음
- 신경망 없음
- LLM(챗봇 같은 거) 없음
- 외부 서버/API 호출 없음

실제로는 **"사람이 미리 정해둔 점수표로 다음 수를 고르는 단순한 봇"** 입니다. 정확한 용어로는 **휴리스틱 봇(heuristic bot)** 또는 **규칙 기반 봇**이에요.

> "AI"는 사용자에게 친숙해서 쓰는 마케팅 용어로 보시면 됩니다. 이 문서에서는 이해를 위해 그대로 "AI"라고 부르지만, 머리속에선 "그냥 봇이구나" 하고 읽어주세요.

---

## 1. 한 줄 요약

난이도는 봇의 **"생각하는 척하는 시간 + 머리 쓰는 방식 + 우선순위"** 세 가지를 바꾸는 것으로 표현돼 있습니다.

---

## 2. 사용자가 보는 흐름

```
앱 실행
  ↓
설정 화면에서 [초급] [중급] [고급] 중 하나 선택
  ↓
"시작" 버튼 누르면 대국 시작
  ↓
내가 한 수 두면
  ↓
봇이 정해진 시간만큼 "생각하는 척"하고
  ↓
선택한 난이도에 맞는 방식으로 다음 수를 결정
```

선택한 난이도 이름은 화면 상단 봇 이름 옆에도 표시됩니다 (예: "AI · 중급").

---

## 3. 비유로 이해하기

세 난이도를 **음식점 셰프**로 비유하면:

| 난이도 | 비유 |
|---|---|
| **초급** | 알바생. 메뉴판도 안 보고 손에 잡히는 재료 아무거나 씀. 단, **자기 음식에 독은 안 넣음**(자기 눈 메우기 같은 자살수만 피함) |
| **중급** | 일반 셰프. 메뉴를 다 훑긴 한데 **시간이 없으면 절반만 보고** 그 안에서 제일 나아 보이는 걸 고름 |
| **고급** | 후보 자리마다 **미리 수천 번 끝까지 둬 보고** 가장 이길 확률이 높은 자리를 고르는 셰프(MCTS). 계산은 별도 주방(웹 워커)에서 하므로 홀(화면)은 멈추지 않음 |

> 셋 다 결국 "사람이 미리 적어둔 규칙대로 점수를 매겨서 골라주는 봇"입니다. 차이는 **얼마나 꼼꼼히 보느냐**일 뿐이에요.

## 4. 실제로 뭐가 달라지나? (3가지 축)

### 축 ① — 생각하는 척하는 시간 (그냥 연출용)

봇이 "고민하는 척"하는 시간입니다. 실제 계산은 거의 즉시 끝나지만, 사용자에게 무게감을 주려고 일부러 기다리게 합니다.

| 난이도 | 대기 시간 |
|---|---|
| 초급 | 0.36초 |
| 중급 | 0.72초 |
| 고급 | 0.15초 대기 + 실제 계산 시간 |

> 초·중급은 그 시간 동안 계산하는 게 아니라 `setTimeout`으로 기다렸다 두는 연출입니다. **고급은 다릅니다** — 대기 후 MCTS가 판 크기별 예산(5×5 0.3초 · 9×9 1.5초 · 13×13 2.5초 · 19×19 4초)만큼 진짜로 계산합니다. 계산은 **웹 워커**(별도 스레드)에서 돌아 그동안에도 화면이 멈추지 않고, 워커를 못 쓰는 환경(예: `file://`로 직접 열기)이면 같은 엔진을 메인 스레드에서 동기 실행합니다.

---

### 축 ② — 다음 수를 고르는 방식 (이게 진짜 차이)

**초급**: 그냥 **랜덤**.
- 가능한 자리들을 다 모은 다음, 그 중에서 무작위로 하나 뽑음.
- 단, **자기 눈을 메우는 자살 같은 수는 거름**.
- → 그래서 초급은 가끔 이상한 곳에 둠. 사람이 봐도 "왜 거기?" 싶은 수.

**중급**: 모든 후보 수에 **점수를 매기고 제일 높은 것을 선택**.
- 점수는 사람이 미리 적어둔 규칙으로 계산함 ("잡으면 +11점", "위험하면 -7점" 등)
- 후보가 너무 많으면(120개 이상) 절반만 골라서 점수 계산. 속도 절약.
- → 합리적인 수를 두지만 가끔 놓치는 부분이 있음.

**고급 (19×19): GNU Go 3.8 (WebAssembly)** — 검증된 공개 엔진을 브라우저에서 직접 구동.
- 순수 MCTS는 19줄 전판 운영이 초심자권이라(실측: GNU Go 최저 레벨에도 전패), 19줄만 GNU Go로 교체. 인터넷 바둑 기준 **약 8~10급**.
- 계산은 웹 워커에서 wasm으로 돌고(수당 평균 ~0.3초), 로드 실패 시 아래 MCTS로 폴백.
- GPL 엔진이라 소스·수정분을 `third_party/gnugo/`에 공개 (자체 대국으로 네이티브 GNU Go와 동급 확인).

**고급 (5·9·13): MCTS(몬테카를로 트리 탐색)** — 점수표와 근본적으로 다른 방식.
- 각 후보 수에서 시작해 **판을 끝까지 무작위로 여러 번 둬 보고**(플레이아웃), 실제로 집을 세서 **이길 확률이 가장 높은 수**를 고름.
- 점수표(휴리스틱)와 달리 **집·사활·전체 판세를 스스로 판단**함 — 이게 실력의 본체.
- 시간 예산 안에서 **수천 번** 시뮬레이션(9×9 1.5초 ≈ 2만 회 · 19×19 4초 ≈ 4천 회). 시뮬을 늘릴수록 강해짐.
- → 점수표가 못 보던 "지금 이 집이 이득인가"를 실제 대국 결과로 판단.
- 큰 판(13·19)은 무작위 탐색이 얕게 퍼지지 않도록 2가지 보정: ① 후보를 **기존 돌 근처 2칸 + 화점**으로 제한(`allowedMask`) ② 잡기/살리기/근접/포석 같은 값싼 휴리스틱을 **사전지식(prior)** 으로 심어 유망수부터 파고듦(`computePriors`).
- **RAVE**: "시뮬레이션 어딘가에서 그 자리에 둔 대국이 이겼나"(AMAF) 통계를 형제 수끼리 공유해, 방문이 적은 초반에도 유망수를 빨리 골라냄. 큰 판일수록 효과 큼.
- **지역성 플레이아웃**: 무작위 대국이 절반 확률로 직전 수 근처(8방)에 응수 → 실제 바둑처럼 접전이 이어져 같은 시뮬 수로 사활·전투 평가가 정확해짐.
- 루트 후보에서 **축(사다리)으로 죽는 도망수는 미리 제거**(`ladderDoomed`) — 무작위 플레이아웃이 십수 수짜리 외길 축을 못 읽는 고전적 약점 보완. 접전 국면 폭발 방지를 위해 검사량 상한(prior 상위 40개·150ms) 있음.
- 검증(자체대국, 같은 시뮬 수): RAVE+지역성 도입 후 **19×19에서 기존 고급에 19/20승(95%)**, 9×9에서 8/9승.

**고급 폴백: 점수표 + 3수 읽기** — MCTS가 패스를 권하거나 엔진을 못 불러온 경우, 그리고 즉답이 필요한 **힌트 버튼**은 이 경로를 씀.
- 모든 수를 평가한 뒤 상위 10개를 **"내가 두면 → 상대 최선 응수 → 내 반격"** 3수 읽음(`bestReplyMove`/`bestReplyScore`).
- **축(사다리) 읽기**: 도망쳐도 끝까지 잡히는 돌은 포기 (`ladderCaptured`).

---

### 축 ③ — 우선순위 (가중치)

중급/고급은 같은 점수 시스템을 쓰지만, **고급은 모든 가중치가 더 과감**합니다.

| 평가 항목 | 중급 이하 | 고급 |
|---|---|---|
| 상대 돌을 잡았을 때 | +11점 | **+16점** (잡기 더 집착) |
| 상대를 단수(아타리)로 몰았을 때 | +3점 | **+5점** |
| 자기 돌이 위험해지는 자리 | -7점 | **-12점** (회피 더 적극적) |
| 상대의 최선 응수 + 내 반격 | **계산 안 함** | **계산함** (2수 더 읽음) |
| 축(사다리) 판정 | **계산 안 함** | **계산함** (죽은 돌 포기) |

> 똑같이 두는 것 같아도 **고급은 더 공격적이고 신중**합니다.

## 5. 그럼 빈 보드에서 첫수는 어떻게? — "위치 자체에 점수가 박혀 있다"

자연스러운 의문이 있어요.

> "점수를 매겨서 고른다고? 그런데 첫수는 잡을 돌도 없고, 주변에 돌도 없는데. 뭘 기준으로 점수를 매기지?"

답: **위치(자리) 자체에 점수가 미리 박혀 있어요.** 봇이 학습한 게 아니라, 만든 사람이 "여기가 좋은 자리야"라고 손으로 가르쳐둔 거예요.

### 항상 켜져 있는 위치 점수 3가지

`evaluateMove()` 안에는 잡기/공격 점수 말고도, **돌이 하나도 없어도 무조건 계산되는** 위치 점수가 있습니다.

**① 가장자리에서 몇 번째 줄인지** (`edgeDist`)
- 3선·4선 → **+2.3점** (집을 짓기 좋은 안정적인 자리)
- 2선 → -1.5점 (애매)
- 1선 → -6점 (가장자리, 거의 쓸모없음)

**② 화점(별점)인지** (`starPointBonus`)
- 화점이면 **+3점**, 아니면 0
- 바둑 격언 "포석은 모서리 → 변 → 중앙" 그대로 박아둔 보너스

**③ 자기 돌의 활로(liberty) 수**
- 활로가 많을수록 좋음. 활로 1개당 +0.9점 (최대 5개까지)
- 중앙 = 활로 4개 → +3.6점 / 코너 = 활로 2개 → +1.8점

### 19×19 빈 보드 첫수 점수 시뮬레이션

| 위치 | 활로 점수 | 가장자리 보너스 | 화점 보너스 | **합계** |
|---|---|---|---|---|
| **소목·화점 (3,3)** | +3.6 | +2.3 (4선) | +3 (화점) | **+8.9** ← 가장 높음 |
| **천원 (10,10)** 정중앙 | +3.6 | 0 (4선 초과) | +3 (화점) | +6.6 |
| **3-3 (2,2)** | +3.6 | +2.3 (3선) | 0 | +5.9 |
| **(1,1)** 2선 | +3.6 | -1.5 | 0 | +2.1 |
| **(0,0)** 1선 코너 | +1.8 | -6 | 0 | **-4.2** ← 페널티 |

→ 그래서 봇은 **첫수로 거의 항상 화점**을 둡니다. 같은 점수의 화점 4개 중 하나는 `Math.random() * 0.25` 노이즈로 무작위로 선택돼요. 매번 같은 자리에 두지 않게 하기 위한 장치.

### 이런 방식의 이름

이걸 일반적으로 **"위치 휴리스틱(positional heuristic)"** 또는 **"오프닝 북(opening book)"** 이라고 부릅니다.

- 알파고도 비슷한 위치 정보를 신경망에 **학습**시켜서 씁니다.
- 이 프로젝트는 학습 없이 **if문 4~5줄로 흉내** 낸 거예요. 그래서 첫수만 보면 사람처럼 두지만, 실력은 초보 수준입니다.

### 그래서 "이런 식의 점수 계산"이 어디까지 통하나?

- **첫수 ~ 초반 ~30수**: 위치 점수가 결정적. 화점/3선·4선 위주로 둠. → 그럴듯해 보임
- **중반**: 잡기·단수·활로 같은 **동적 점수**가 위치 점수를 압도. 이때부터 실력 차이가 드러남
- **종반(끝내기)**: 한 수 앞만 보는 단점이 부각됨. 큰 그림을 못 봄

> 즉, 위치 점수는 **"초반에 그럴듯하게 두는 척"** 하는 데 효과적이고, 중반 이후는 잡기 위주의 단순한 봇 본색이 드러난다고 보시면 됩니다.

## 6. 정리: 한눈에 보는 차이표

| | **초급** | **중급** | **고급** |
|---|---|---|---|
| 알고리즘 | 랜덤 | 점수 평가 | **19줄 GNU Go(WASM) / 5~13줄 MCTS** (웹 워커) |
| 집·사활 판단 | X | X (점수표 근사) | **O**(MCTS가 실제 대국으로) |
| 상대 응수 읽기 / 축 판정 | X | X | **O** |
| 잡기/공격 성향 | 낮음 | 중간 | **높음** |
| 응답 속도 | 빠름 (0.36초) | 보통 (0.72초) | 실제 계산 (5×5 0.3초 ~ 19×19 4초) |
| 추천 대상 | 처음 두는 사람 | 룰은 아는 사람 | 어느 정도 둘 줄 아는 사람 |

## 8. 한계 / 발전 방향

초급/중급은 **사람이 손으로 적은 점수표**가 전부이고, 고급(MCTS)도 신경망 없는 무작위 시뮬레이션이라 천장이 있습니다.

- 알파고처럼 학습한 모델이 아니라서 **창의적이거나 깊은 수를 못 둠** — MCTS의 이론적 천장은 아마추어 고급 수준
- 실측(2026-07, GNU Go 3.8과 19×19 맞바둑): MCTS는 시뮬 4,500회/수 예산으로 GNU Go(약 8~10급) 상대 전패 — **MCTS의 19×19 전판 운영은 초심자권(15급 이하)**. 이 실측이 19줄을 GNU Go(WASM)로 교체한 이유. 작은 판(9×9)은 시뮬 밀도가 20배라 MCTS로도 체감 8~10급
- 초급/중급은 한 수 앞만 보는 "욕심쟁이" 방식. 고급 폴백(힌트 등)도 **3수까지만** 읽음 — 축은 예외적으로 끝까지 읽음
- 같은 상황에서 매번 똑같이 두지 않게 약간의 무작위성만 더해놓음 (점수표는 `Math.random() * 0.25` 노이즈, MCTS는 시뮬 자체가 무작위)

진짜 강한 바둑 AI를 만들고 싶다면:
1. **더 깊은 수읽기** (미니맥스, 알파-베타 가지치기)
2. ~~**몬테카를로 트리 탐색(MCTS)**~~ → **고급에 도입 완료** (2026-07). 처음엔 작은 판(5·9)만이었지만, 후보 제한(`allowedMask`)+사전지식(`computePriors`)으로 큰 판(13·19)까지 확대(자체대국에서 기존 3수읽기 고급을 63% 이김). 이후 **RAVE + 지역성 플레이아웃**을 더해 19×19에서 직전 버전을 95%(19/20) 이김. 시뮬을 늘릴수록 강해지는 게 강점(점수표 튜닝은 천장이 있음).
3. **신경망 학습** — 수많은 기보로 학습시키기 (알파고 방식, 이 앱엔 오버킬)

남은 개선 여지:
- **시뮬 예산 상향**: 기기 성능에 맞춰 예산을 늘리면 더 강해짐 (지금은 5×5 0.3초 ~ 19×19 4초 고정)
- **플레이아웃 정책 고도화**: 지금은 "단수 처리 + 지역성 응수(직전 수 8방) + 무작위". 점수표까지 플레이아웃 정책으로 쓰면 같은 시뮬 수로 더 정확해짐
- **트리 재사용**: 매 수 트리를 버리고 새로 탐색 → 상대가 예상 수를 두면 서브트리를 이어 쓰면 공짜 시뮬 확보

지금 구조의 강점은 여전히 **빠르고 가볍고 서버가 필요 없다**는 점입니다. MCTS도 100% 브라우저에서 도는 순수 JS라 Vercel 정적 배포 구조를 그대로 씁니다.
